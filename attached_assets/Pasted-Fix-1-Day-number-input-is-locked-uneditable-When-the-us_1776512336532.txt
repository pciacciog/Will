Fix 4 — Milestone notifications must actually fire on the correct day
This is the most critical fix. The milestones are being stored in wills.milestones as a JSON array but confirm that checkMilestoneNotifications() in scheduler.ts is correctly reading them and firing on the right day.
Find checkMilestoneNotifications() and verify:
Day calculation:
typescriptconst streakDay = differenceInDays(now, will.streakStartDate) + 1;
This must match what the user configured — if user set Day 3, the notification fires when streakDay === 3.
Timing: the scheduler runs every minute. Milestone notifications for a given day should fire once, at a random time during that day. Confirm dedup via sentMilestones is working — once a milestone day fires, its day number is added to sentMilestones JSON array and it never fires again.
If the Will slips (streak resets): sentMilestones is cleared to [] and streakStartDate is reset to NOW(). This means all milestones become eligible again from the new streak start. Confirm this is happening correctly in resetAbstainStreak().
Random time within the day: currently the scheduler fires as soon as it detects the streak day matches. Add a simple randomization — when a milestone day is first detected, store a random fire time for that day (between 9am and 8pm user local time) and only fire when now >= randomFireTime. This prevents all milestones from firing at midnight on their trigger day.
If the random fire time approach is complex to implement, a simpler acceptable alternative is to fire the milestone notification between 9:00 AM and 9:30 AM on the trigger day — check if now.hour === 9 and streakDay === milestone.day and not yet in sentMilestones.
