Today Screen — Personal / Work Toggle

Concept
The Today screen currently shows one flat list of intentions. We are adding a Personal / Work toggle at the bottom of the screen so users can keep their personal and work intentions completely separate. Same screen, same UI, same Today/Tomorrow toggle at the top — just two isolated contexts switchable from a fixed footer toggle.
This is a context switch, not a filter. Personal intentions and Work intentions are entirely separate lists. Adding something in Personal never appears in Work and vice versa. Please refer to the screenshots attached footthe updated design. Also, please capitalize the letter “w” in “I will:” title text. Should read: “I Will:”

The existing "‹ Home" floating text nav needs to be replaced with a proper structured nav bar matching the rest of the app: use the same structured back button that is already present on other screens.

Data model
Add a context field to the intentions/today items table:


ts
context: text("context").default("personal").notNull() // 'personal' | 'work'
Run npx drizzle-kit push after updating the schema.
All existing intentions without a context value default to 'personal' automatically via the column default.

State
Add a context state to Today.tsx:


ts
const [context, setContext] = useState<'personal' | 'work'>('personal');
Persist this to localStorage so the user's last selected context is remembered between sessions:


ts
const [context, setContext] = useState<'personal' | 'work'>(() => {
  return (localStorage.getItem('today_context') as 'personal' | 'work') ?? 'personal';
});

useEffect(() => {
  localStorage.setItem('today_context', context);
}, [context]);

Data fetching
Pass context as a query parameter when fetching today's intentions:


ts
const { data } = useQuery(
  ['intentions', activeDate, context],
  () => fetchIntentions({ date: activeDate, context }),
  { staleTime: 5 * 60 * 1000 }
);
The query key includes context so switching the toggle automatically loads the correct list without any manual refresh.

Backend changes
GET /api/intentions
Accept an optional context query param (personal | work). Filter results by context. If no context param is provided, default to personal for backwards compatibility.
POST /api/intentions
Accept context in the request body. Store it on the intention record. Default to personal if not provided.
PATCH /api/intentions/:id
No change needed — context is set at creation and never changes.

Checklist behavior per context
The Today/Tomorrow toggle at the top operates independently per context:
* Personal·Today, Personal·Tomorrow, Work·Today, Work·Tomorrow are four completely independent lists
* Switching context while on Tomorrow stays on Tomorrow — the two toggles are independent axes
The midnight reset applies to both contexts — both Personal·Today and Work·Today reset at midnight.
The AddIntentionInput (already extracted) submits with the current active context value passed as a prop:


tsx
<AddIntentionInput
  onAdd={(text) => createIntention({ text, context })}
  placeholder={context === 'work' ? 'Add a work intention...' : 'Add another...'}
/>

Visual feedback on context switch
When the toggle is tapped, the active checked item indicators follow the context color:
* Personal context: checked items use green (#1a9e75) checkbox fill
* Work context: checked items use blue (#4a7fd4) checkbox fill
This gives the whole list a subtle contextual identity beyond just the toggle itself.
