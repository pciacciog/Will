DATABASE SCHEMA:

**New table for check-ins:**
```sql
will_check_ins (
  id SERIAL PRIMARY KEY,
  will_id INTEGER REFERENCES wills(id),
  user_id INTEGER REFERENCES users(id),
  date DATE, -- The day being tracked (e.g., Jan 22)
  status VARCHAR, -- 'yes', 'no', or 'more_or_less'
  reflection_text TEXT, -- Optional
  checked_in_at TIMESTAMP, -- When they actually logged it
  is_retroactive BOOLEAN DEFAULT false, -- Was this filled in late?
  created_at TIMESTAMP DEFAULT NOW()
)
```

**Update wills table:**
```sql
ALTER TABLE wills ADD COLUMN check_in_type VARCHAR DEFAULT 'one-time';
-- 'daily' or 'one-time'
```

**Optional: Store final reflection for daily Wills:**
```sql
will_reflections (
  id SERIAL PRIMARY KEY,

  user_id INTEGER REFERENCES users(id),
  feeling VARCHAR, -- 'great', 'okay', or 'could_improve'
  final_thoughts TEXT,
  created_at TIMESTAMP DEFAULT NOW()
)
```

RETROACTIVE CHECK-INS (FILL IN MISSING DAYS):

**Key Feature:** Users can go back and fill in days they forgot to check in.

**During the Will:**
- If user missed checking in on Tuesday, they can go back on Thursday and fill it in
- Show "Fill In Missing Days" button when there are gaps
- Let them select which day to fill
- Same check-in modal appears

**At the End:**
- Show all days with their status
- Allow editing ANY day (missing or already completed)
- User confirms data is accurate before continuing

**Implementation:**
- Check-ins store the `date` being tracked + `checked_in_at` timestamp
- Can have Tuesday's check-in logged on Thursday (retroactive)
- `is_retroactive` flag for analytics (optional)

SUCCESS RATE CALCULATION:

For daily check-ins:
Success Rate = (Days marked "Yes") / (Total check-ins completed)
Example: 7 days total
* 5 days: Yes
* 1 day: No
* 1 day: More or Less
Success Rate = 5/7 = 71%
Or include "More or Less" as partial success: Success Rate = (5 + 0.5) / 7 = 78%
